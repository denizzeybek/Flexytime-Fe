# Internationalization (i18n)

`vue-i18n` v9 (non-legacy). Every user-facing string goes through it.

---

## 1. Setup in this repo

- Plugin: `src/plugins/i18n.ts` — creates the i18n instance, registers `en` eagerly, lazy-loads `tr`.
- Locale files: `src/locales/en.json` and `src/locales/tr.json`.
- The `en.json` type is exported as `MessageSchema` and used to type `useI18n`.
- Locale switching is handled by `setI18nLanguage(locale)` which also swaps dayjs locale and sets `Accept-Language`.
- Default locale: `en` (`VITE_I18N_LOCALE`), fallback: `en`.
- **Turkish is a first-class locale** — never ship an English-only key.

---

## 2. How to use it

### In templates — `$t`

```vue
<h1 class="text-2xl font-bold text-content-primary">
  {{ $t('pages.auth.login.title') }}
</h1>
<p class="text-sm text-content-secondary">
  {{ $t('pages.auth.login.subtitle') }}
</p>
```

### In `<script setup>` — `useI18n` with `MessageSchema` generic

This is the repo convention. Always type `useI18n` with `MessageSchema` for autocompletion and compile-time safety:

```ts
import { useI18n } from 'vue-i18n'

import { type MessageSchema } from '@/plugins/i18n'

const { t, locale } = useI18n<{ message: MessageSchema }>()

const title = t('pages.auth.login.title')
```

### Interpolation — never concatenate

```ts
// ✅
t('greeting.hello', { name: user.name })
// messages: "Hello, {name}!" / "Merhaba, {name}!"

// ❌
t('greeting.hello') + ' ' + user.name
```

### Pluralization

```json
// en.json
{ "items": "{count} item | {count} items" }
// tr.json
{ "items": "{count} öğe | {count} öğe" }
```

```ts
t('items', { count: 3 }, 3)
```

## 3. Key structure — match the existing tree

Keys are `dot.case`, hierarchical by feature. The top-level sections in `en.json`:

```
language
languages
pages.<feature>.<view>.<section>.<key>
  e.g. pages.auth.login.form.email.label
common.<scope>.<key>
  e.g. common.validation.mixed.required
  e.g. common.validation.fields.task
components.<componentName>.<key>
  e.g. components.input.noOptionFound
```

### Rules

- **Feature-level strings** → `pages.<feature>.<view>.<key>`.
- **Reusable messages** (validation, confirm dialogs, "save"/"cancel") → `common.*`.
- **Strings owned by a shared component** → `components.<componentName>.*`.
- Match the **depth and nesting of the existing tree** — if the sibling view uses `form.<field>.label` / `form.<field>.placeholder`, mirror it exactly.
- Key names are `camelCase` inside a segment: `forgotPassword`, `createAccount`, `googleLogin`.

### Before adding a key, grep

1. Open `src/locales/en.json`.
2. Search for the English phrase (or close variants).
3. If it exists — reuse.
4. If not, add to the **smallest** matching scope (prefer `components.*` or `common.*` over duplicating under every feature).

## 4. Adding a new key — the checklist

You must add to **both** files. Untranslated keys are a blocker.

```jsonc
// src/locales/en.json
{
  "pages": {
    "timesheets": {
      "entries": {
        "saveButton": "Save entry"
      }
    }
  }
}

// src/locales/tr.json — SAME PATH, Turkish value
{
  "pages": {
    "timesheets": {
      "entries": {
        "saveButton": "Kayıt oluştur"
      }
    }
  }
}
```

Then:

- [ ] Key exists in `en.json`.
- [ ] Key exists in `tr.json` at the **same path** with a **real** Turkish translation (no placeholders, no copied English).
- [ ] No stale/orphan keys left behind after a rename. Grep the old key across the codebase and purge.
- [ ] `yarn dev` renders both locales without `[intlify]` missing-key warnings in the console.

## 5. Validation messages (vee-validate + yup)

The repo uses two patterns — both are valid, pick the one that matches the surrounding file.

### Pattern A — `.label()` with i18n (preferred for simple forms)

Yup picks a default message based on the validator, `.label()` injects the localized field name:

```ts
const validationSchema = object({
  email: string()
    .email()
    .required()
    .label(t('pages.auth.login.form.email.label')),
  password: string()
    .required()
    .label(t('pages.auth.login.form.password.label')),
})
```

### Pattern B — explicit `common.validation.*` keys (for fields with interpolation)

```ts
const validationSchema = computed(() =>
  object({
    taskName: string()
      .required(t('common.validation.mixed.required', { field: t('common.validation.fields.task') }))
      .label(t('common.validation.fields.task')),
  }),
)
```

Note: when labels or messages depend on reactive i18n (locale switch), wrap the schema in `computed(() => …)` so it updates.

### Don'ts

- ❌ Hardcoded strings in validation: `.required('Email is required')`. Always a key.
- ❌ Inline English-only fallback: `.required(t('…') ?? 'Fallback')`. If the key exists in both files, no fallback is needed.

## 6. Toasts, dialogs, aria

All of these are user-facing — all go through `t(…)`:

```ts
const { showSuccessMessage, showErrorMessage } = useFToast()

showSuccessMessage(t('timesheets.entries.saved'))
showErrorMessage(t('common.errors.generic'))
```

```vue
<button :aria-label="$t('common.actions.close')" />
```

---

## 7. Routes titles

Route `meta.title` / `meta.name` in this repo store an `ERouteNames` enum value (a display label, not an i18n key — see `router/routeNames.enum.ts`). If you need the localized tab title, wire it through i18n in the layout / `useRouter` watcher, not in `meta`. Don't change this convention unilaterally.

---

## 8. Common mistakes — reject on sight

- 🚫 Hardcoded English/Turkish string in any `.vue` template.
- 🚫 Placeholder, label, empty state, aria-label, tooltip, toast, validation message as a raw string.
- 🚫 Key added to `en.json` only.
- 🚫 Copy of the English value in `tr.json`.
- 🚫 `String concatenation` to build a message.
- 🚫 Invented keys like `'auth.login.forgot'` when the existing tree uses `'pages.auth.login.forgotPassword'`.
- 🚫 Using `useI18n()` without the `<{ message: MessageSchema }>` generic in new code.
